# 安全审查方法论

上线前最后一关。扫描常见安全漏洞，确保产品交付安全。

## 审查清单

### 🔑 认证安全
- [ ] 密码是否哈希存储？（不能用明文/base64）
- [ ] 登录有无暴力破解限制？（限流/验证码）
- [ ] Token/JWT有过期时间吗？
- [ ] 密码重置流程安全吗？（验证码+时效）
- [ ] 是否支持多设备登录互踢？

### 🔐 权限安全
- [ ] 普通用户能访问管理API吗？
- [ ] API端点有权限校验吗？还是在中间件层统一处理？
- [ ] 用户能查看/修改其他用户的数据吗？（水平越权）

### 💰 支付/资金安全
- [ ] 订单金额能在客户端修改吗？
- [ ] 支付回调有签名验证吗？
- [ ] 重复支付通知会被多次处理吗？（幂等性）
- [ ] 退款逻辑正确吗？

### 🛡️ 数据安全
- [ ] API Key/密钥存在代码里还是环境变量？
- [ ] 敏感信息返回到前端了吗？
- [ ] 日志里会打印密码/Token吗？
- [ ] 数据库连接信息泄露了吗？
- [ ] 用户数据导出包含其他用户信息吗？

### 🌐 前端安全
- [ ] 所有用户输入有转义吗？（XSS防护）
- [ ] API返回的HTML安全渲染了吗？
- [ ] CORS配置收紧了吗？（只允许特定域名）
- [ ] CSP头设置了没？

### 🚪 配置安全
- [ ] 默认密码改了吗？
- [ ] Debug模式关了没？
- [ ] 错误信息不暴露堆栈给用户
- [ ] 无用的管理端口关了吗？

## 评分等级

```
P0 (立即修) — 资金/用户数据/越权漏洞
P1 (尽快修) — XSS/SQL注入/信息泄露
P2 (排期修) — 缺少安全头/CORS过松
```
