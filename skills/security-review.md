# Security Review — 安全审查

## 适用场景
上线前、大版本发布前、接手遗留系统时。

## 审查清单

### 🔑 认证与授权
- [ ] 密码哈希存储（不用 MD5/base64）
- [ ] 登录有频率限制（5 次/分钟 = 封验证码）
- [ ] Token/JWT 有过期时间
- [ ] 各 API 端点有权限校验
- [ ] 不存在水平越权（用户A看到用户B数据）

### 💰 资金安全
- [ ] 订单金额不可在客户端篡改
- [ ] 支付回调有签名验签
- [ ] 幂等处理（重复通知不重复入账）
- [ ] 退款逻辑闭环

### 🛡️ 数据安全
- [ ] 密钥存在环境变量，不在代码里
- [ ] 敏感信息不返回前端
- [ ] 日志不打印密码/Token
- [ ] 用户数据导出不包含他人信息
- [ ] 输入有转义（防 XSS/SQL 注入）

### 🌐 前端安全
- [ ] CORS 白名单收紧
- [ ] CSP 头部设置
- [ ] iframe 有 sandbox 限制

### ⚙️ 配置安全
- [ ] 默认密码已改
- [ ] Debug 模式已关
- [ ] 错误不暴露堆栈给用户
- [ ] 无用端口/服务已关闭

## 输出格式

```
✅ 通过 | ⚠️ 注意 | ❌ 未通过

❌ [P0] API 端点无认证
  位置: /api/admin/users
  风险: 任何人可访问用户列表
  修复: 加 JWT 中间件验证
```

## 常见高危模式

```
# P0 — 看到直接标记
admin_auth = False          # 调试用，上线必须删
password = "123456"         # 硬编码密码
eval(request.body)          # 代码注入
"SELECT * FROM users WHERE id = " + uid  # SQL 注入
```
