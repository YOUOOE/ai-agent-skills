# 代码审查方法论

系统化代码审查流程，确保代码质量、安全性和可维护性。

## 审查清单

### 逻辑正确性
- [ ] 边界条件处理了没？（空值/0/null/超大值）
- [ ] 异常分支都走了吗？（try-except范围正确吗）
- [ ] 并发安全？（共享变量加锁了吗）
- [ ] 幂等性？（重复执行结果一致吗）

### 安全性
- [ ] 用户输入有校验/转义吗？（SQL注入/XSS）
- [ ] API Key/密钥硬编码了吗？
- [ ] 权限检查到位了吗？（用户不能越权）
- [ ] 敏感信息会泄漏到前端/日志吗？

### 可维护性
- [ ] 函数职责单一吗？（一个函数只做一件事）
- [ ] 命名清晰吗？（不用a/b/c/tmp这种）
- [ ] 有重复代码可以抽取吗？
- [ ] 类型注解完整吗？

### 性能
- [ ] 数据库查询有N+1问题吗？
- [ ] 循环里有昂贵的重复计算吗？
- [ ] 大文件/大数据有分批处理吗？

### 回归风险
- [ ] 改动的模块有其他调用方吗？
- [ ] 存量的旧数据兼容吗？
- [ ] 配置项改了会影响其他环境吗？

## 审查流程

```
拿到PR → 先看描述理解意图
   ↓
读改动文件 → 按清单逐项检查
   ↓
标注问题 → P0(必修) / P1(建议) / P2(可选)
   ↓
写审查意见 → 问题+根因+修改建议
   ↓
验证修复 → reviewer确认后approve
```

## 输出格式
```
[P0] 安全问题：API Key硬编码在配置文件
  位置: config.py:15
  建议: 改用环境变量读取

[P1] 逻辑问题：除零未处理
  位置: calculator.py:42
  建议: 加除零判断
```
