# 系统化调试方法论

系统性定位Bug根因的工作流，从"看到了什么"到"为什么发生"。

## 方法论

### Phase 1 — 复现
- 精确记录复现步骤
- 确认环境一致性（生产vs开发）
- 确认问题的可复现率和触发条件

### Phase 2 — 根因分析
- **二分法定位**：如果有多步操作，从中间开始排查
- **最小化测试**：写最小可复现代码，排除干扰因素
- **追踪数据流**：输入→处理→输出，每一步验证
- **对照有效路径**：之前能用的版本变了什么？diff对比

### Phase 3 — 多路径推演
一个Bug通常有3+种可能根因，列出并排序：
1. 最可能路径（代码逻辑错误）
2. 第二可能路径（数据状态不一致）
3. 第三可能路径（环境/依赖问题）
按顺序验证，不跳过。

### Phase 4 — 最小修复
- 只改根因，不顺手改无关代码
- 改完立即验证修复有效
- 再验证关联功能不受影响

### Phase 5 — 闭环
- 写一条教训：根因是什么？怎么发现的？怎么修的？
- 检查同样的Bug模式是否在别处存在

## 陷阱
- ❌ 不看Trace直接猜根因
- ❌ 修了一个表象就停，不追根因
- ❌ 改完不验证关联功能

## 示例：前端页面白屏
```
复现 → 打开页面空白
二分 → 检查HTML渲染→正常，JS执行→报错
缩小 → 定位到某个script块的SyntaxError
根因 → 模板字符串里嵌套了反引号导致语法错误
修复 → 改用拼接方式或转义
验证 → node --check + 浏览器打开确认
闭环 → 写教训：HTML内嵌JS修改后必须做语法检查
```
