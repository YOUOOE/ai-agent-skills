# Self-Evolution — 自适应优化框架

## 适用场景
想让 AI Agent 能自主发现能力短板→实践改进→固化经验。适合追求 Agent 持续自我优化的团队。

## 核心理念

一个让 AI 持续发现自身不足并尝试改进的设计框架。核心思路：不断回顾 → 发现问题 → 尝试改进 → 验证效果。目前是方法论层面的设计，适合作为 Agent 自我优化的参考框架。

## 8 阶段循环

### Phase 1 — Review（回顾）
扫描近期的操作日志和结果记录，提取以下信号：
- 反复出现的错误或失败模式
- 用户多次纠正或提醒的点
- 明显做不好/不会做的任务类型
- 重复劳动（每次都要做的事）

### Phase 2 — Ideate（构思）
基于实际数据提出改进方案。每个方案涵盖：
- 观察到什么问题
- 问题为什么会发生
- 改哪里能解决
- 怎么确认改对了

### Phase 3 — Modify（修改）
- 每次只改一个点
- 涉及文件不超过 5 个
- 改动前先备份原始版本

### Phase 4 — Commit（记录快照）
改完先保存版本快照，再做验证。记录信息包含：
- 改了哪个文件、哪个位置
- 基于什么观察做出的修改
- 期望解决什么问题

### Phase 5 — Verify（验证）
分层验证：
- L1：语法检查和结构完整性
- L2：功能验证（实际运行/接口测试）
- L3：回归检查（改了 A，B/C 是否正常）

### Phase 6 — Gate（把关）
5 项检查全部通过才算成功：
- 安全（没有引入新风险）
- 稳定（现有功能不受影响）
- 有效（确实解决了问题）
- 性能（没有明显变慢）
- 规范（代码风格保持统一）

任一不通过 → 自动恢复到修改前的版本。

### Phase 7 — Log（记录沉淀）
把这次改进的经验写入知识库：
- 修了什么
- 根因是什么
- 怎么验证的
- 后续怎么预防

### Phase 8 — Loop（下一轮）
回到 Phase 1，继续发现下一个可以优化的地方。

## 原则

1. **每次只改一个地方**，不批量改
2. **版本管理控制流程**，改坏了就恢复
3. **基于实际数据做判断**，不靠猜测
4. **通过验证才算完成**，不能自己说改好了就是好了
